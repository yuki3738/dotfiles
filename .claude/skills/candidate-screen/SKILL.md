# 候補者スクリーニング

候補者のプロフィール情報と `hr/positions/` 配下の求人票を照合し、カジュアル面談を実施すべきかを判断する。

## 入力

以下のいずれかの形式で候補者情報を受け取る：

1. **URL**: 候補者のプロフィールページURL（採用媒体、LinkedIn等）
2. **テキスト**: 履歴書・職務経歴書の内容をそのまま貼り付け

## 処理の流れ

### Step 1: 候補者情報の取得

- URLが渡された場合: ブラウザツール（mcp__claude-in-chrome）またはWebFetchで候補者プロフィールを取得する
  - ログインが必要なページの場合はmcp__claude-in-chrome を優先する
  - ページが取得できない場合はユーザーにテキスト貼り付けを依頼する
- テキストの場合: そのまま使用する

候補者の以下の情報を整理する：
- 技術スキル・使用言語/フレームワーク
- 業務経験・経歴
- 志向性・関心領域（読み取れる範囲で）
- **勤務条件の希望**（週4勤務・時短勤務・副業前提などの記載がないか必ず確認する）

#### Step 1-A: US面談情報の取得（マッチページの場合は必須）

マッチページ（`/matches/xxxx`）の場合、プロフィール情報だけでなく *US面談情報セクションの確認を必ず最初に行う*。US面談情報はFindy担当者（US）が候補者と面談して得た第三者評価・裏情報を含んでおり、プロフィールだけでは読み取れない以下の重要情報が記載されていることが多く、判断を大きく左右する。

- 日本語コミュニケーション能力の第三者評価（「◎」「ビジネスレベル」等）
- 実際の希望勤務条件（出社可否・リモート度合い・時短希望の有無・副業意向）
- 転職検討理由の本音（プロフィール自己PRよりも率直に書かれる）
- 希望年収の下限・上限、交渉余地
- 性格・協調性・リーダーシップ等の人物評価
- 短期離職・ブランク・外国籍候補者の在留状況など、レッドフラグの補足説明

取得手順:

1. マッチページを`get_page_text`または`read_page`で取得する
2. *ページ内に「US面談情報」セクションが存在するか必ず確認する*（`get_page_text`だけでは本文が省略されている場合がある。`read_page`で"US面談情報"見出し・「Findy USからの申し送り事項」リージョンを探す）
3. 存在する場合、*「▼全文を表示する」ボタンをクリックしてから再取得*する（初期表示は数十文字で切れているため、全文展開しないと本文の大半が欠落する）
4. 展開後のテキストを`get_page_text`または`screenshot`＋`read_page`で取得し、プロフィール情報と合わせて判断材料とする

US面談情報が存在する場合、出力の「候補者サマリー」または「強み／懸念点」にその内容を必ず反映すること（特に担当USの所感・希望勤務条件・転職理由）。US面談情報を見落としたまま判断すると、外国籍候補者の日本語力や短期離職の背景などを誤って評価するリスクがある。

US面談情報が存在しない求人（企業独自スカウト由来など）や、候補者詳細ページ（`/candidates/xxxx`）を直接指定された場合はこのステップはスキップする。

#### Step 1-B: 勤務条件の確認

プロフィール本文だけでなく、自己PR・希望条件・US面談メモなどに週4勤務・時短希望・副業前提・リモート限定・海外在住などが書かれていないか見落とさず検出する。これらは出力の「確認事項」としてユーザーに提示すること。当社の勤務制度で受け入れ可能かどうかはAIが判断せず、必ずユーザーに確認を取る。

### Step 2: 求人票の読み込み

以下の優先順位で求人票を取得する：

1. `hr/positions/` 配下の全求人票（*.md、templates/とshared/は除く）を読み込む
2. `hr/positions/` が存在しない場合は、マッチページに紐づいている求人票URLにブラウザでアクセスして内容を取得する
3. どちらも取得できない場合はユーザーに求人票の提供を依頼する

各求人票から以下を抽出する：
- 必須スキル
- 歓迎スキル
- 求める人物像

### Step 3: 照合と判断

各求人ポジションについて、候補者の情報と照合する。

**判断基準:**
- 必須スキルの充足度が最重要。必須スキルの大半を満たさない場合はミスマッチと判断する
- 歓迎スキルの充足は加点要素として扱う
- 求める人物像は、経歴の方向性や記載内容から推測できる範囲で判断する（確証がない場合は「面談で確認」とする）
- カジュアル面談はカジュアルな場なので、明らかなミスマッチでない限りは「面談すべき」寄りの判断でよい

**総合評価の3軸（「良い点 vs 懸念点」の個数比較ではなく、3軸の総合点で見る）:**

「スキル幅が広い」「AI活用積極的」など個別の強みだけを見て面談推奨に倒すのは危険。以下の3軸で総合評価し、*複数軸で懸念側に倒れている場合は個別の強みでは覆せない*と判断する。

1. *スキルの深さ・一貫性*
   - 技術幅の広さだけでなく、*どの技術・領域を「自分の武器」として深く掘ったか*（T字の縦軸）が履歴から読み取れるか
   - ロール（メンバー／リーダー／テックリード／EM／CTO）がプロジェクトごとに散らばっていないか
   - 「10年の経験」と「1年の経験を10回繰り返している」を見分ける。在籍期間×同一領域の連続性で判断
   - 幅広いがどこも数ヶ月〜1年で次の領域に移っているなら、シニア級の実力があるとは評価しにくい

2. *勤務条件の整合性（個別ではなく「束」で見る）*
   - 時短・副業OK・リモート可・特定地域限定・希望年収高などを *1項目ずつ個別に足切り*するのではなく、*複数重なったときにフルコミットの絵が描けるか*で判断する
   - 特に「副業OK × 時短制度あり」の組み合わせは、本業稼働を絞る意図が透けており、当社のフルタイム・プロダクトコミット前提とミスマッチ
   - 「当社で正社員として長期で価値を出してくれる絵」が描けるかを総合評価する

3. *組織コミット志向*
   - 業務委託中心の履歴か、正社員で長期コミットしてきた履歴か
   - 複数社並行の業務委託期間がないか
   - 短期離職（在籍1年未満で転職、入社直後に転職活動中）のパターンが継続的でないか
   - 「組織に属してプロダクトを育てる」志向か、「案件ベースで腕を売る」志向かを見極める

3軸のうち2つ以上が懸念側に倒れている場合は、個別の技術的魅力があっても基本お見送り。「技術的には面白そうだが面談する価値があるか」の判断軸はスキル×勤務条件×コミット志向の総合点である。

**経歴の整合性チェック（レッドフラグ検出）:**

必須スキルの充足だけでなく、以下の観点で経歴の整合性を確認する。複数該当する場合はお見送り寄りの判断とする。

- **経験年数と年収の乖離**: 経験年数が長いのに現年収・希望年収が市場水準より著しく低い場合、スキルの質や成果に疑問がある。例：エンジニア経験8年で年収300万未満は要注意
- **雇用形態の下降**: 正社員→契約社員・業務委託へのダウンは、パフォーマンスや職場適応に課題があった可能性を示唆する
- **卒業後・退職後のブランク**: 説明のつかない長期ブランク（1年以上）がある場合は注意
- **自己申告スキルの水増し感**: 多数のツール・言語を列挙しているが、実務で使用した記載があるのはごく一部の場合、スキルの深さに疑問がある
- **在籍期間に対して成果記載が薄い**: 長期間在籍しているのに具体的な成果・改善実績の記述が少ない場合、主体的に動いていなかった可能性がある
- **希望年収と求人レンジの乖離**: 希望年収が求人レンジの下限を大きく下回る場合、求人が求めるレベル感と候補者の自己評価にギャップがある可能性。単純に「年収面でミスマッチリスクが低い」とポジティブに解釈しない

これらは単独では判断材料にならないこともあるが、複数重なった場合は総合的にお見送りと判断する。出力時は「懸念点」セクションにレッドフラグとして明記すること。

**年齢 × 実務経験年数 × 年収の三角測量:**

「経験年数と年収の乖離」だけでなく、*年齢も加味して市場価値を判定する*。年齢から逆算した「同年代の他エンジニアの平均的なキャリア蓄積」と比較してギャップが大きい場合はレッドフラグ。

- 30代後半以降で実務エンジニア経験が5年未満の場合、*キャリア後発組（前職が非エンジニア・遅咲き転職組）* の可能性が高く、同年代と比べて設計判断・技術選定・リード経験で差がつきやすい
- *年齢相応の年収に達していない*場合（例：35歳で500万未満、40歳で500-600万未満）、スキルの質・市場での評価が低い証左。例：40歳でフロントエンド実務4年・現年収300-350万・希望500万 → キャリア蓄積も年収も同年代水準を下回るレッドフラグ複合
- 「シニア・テックリード級を求める求人」に対して、**年齢40歳以上で実務経験5年未満かつ年収が市場下位の候補者は基本お見送り**

**ポジションの「手練れ度」と候補者キャリアの整合性:**

求人票の必須スキルに以下が含まれる場合、その求人は *手練れ前提（設計リード級以上）* と判断する。

- 「アーキテクチャ設計・技術選定への主体的関与」
- 「リードエンジニア・テックリード経験」
- 「コードレビューによる品質向上への貢献」（チーム規模を主導する役割として）
- 「保守性・パフォーマンスを考慮した設計・実装」（単なる実装ではなく設計責任を含む文脈で）

これらの求人に対して *メンバーポジションを継続している（複数年在籍してもメンバー止まりの）候補者は基本ミスマッチ*。設計リード経験の有無を必ず確認し、無ければ見送り判断に倒す。「実務年数は満たしている」だけで判断しない（手練れ求人は質的要件であり、量的要件ではない）。

**勤務地・リモートの要件（全ポジション共通）:**
- 国内フルリモートは制度上可能。国内在住であれば勤務地は懸念点にならない
- ただし**フルリモート希望を全面に出している候補者はマイナス評価**とする。出社やチーム対面への柔軟性が低い可能性があるため、懸念点として記載すること
- **ジュニア卒業レベル（実務3年前後）でフルリモート前提を要求する候補者は基本お見送り**。シニア級なら成果ベースでフルリモートも検討可能だが、ジュニア〜ミドルは対面でのオンボーディング・レビュー・メンタリングが成長に直結するため、フルリモート前提は本人にとっても組織にとっても望ましくない
- **海外在住の候補者は、特別な理由がない限り基本お見送り**。時差だけでなく、法務・労務面のハードルが高いため海外就業そのものがNG

**勤務形態の要件（全ポジション共通）:**
- **時短勤務希望は基本お見送り**。当社の求人はフルタイム前提で時短勤務制度を設けていない
- 候補者の希望条件に「時短勤務制度あり」が含まれている場合、他の条件にかかわらず基本お見送りと判断する

**訪日ラボ（インバウンド支援事業本部）固有の勤務要件:**
- **原則出社**。家庭の事情等でリモートすることは許容されるが、基本は出社が前提
- 候補者の希望条件に「リモートワーク可」が含まれている場合、訪日ラボのポジションではマイナス要素として扱う
- 訪日ラボのポジション（2203等）にマッチングする際は、出社可能かを必ず確認事項に含めること

**訪日ラボ ジュニアエンジニア採用ペルソナ:**
- 訪日ラボではジュニアエンジニアの採用も行っている。ペルソナは `2-Resources/recruit/positions/personas/hounichilab-junior-engineer.md` を参照
- パターン1「自走型自動化オタク層」: Web広告代理店/事業会社の運用担当出身（25〜27歳、社会人3年目付近）で、GAS等による業務自動化の実績があり、エンジニア転身を志向する人材。MARCH・関関同立以上の地頭の良さ、ビジネスサイドとのコミュニケーション力、自走力が重要な評価軸
- このペルソナに合致する候補者は、エンジニア実務経験がなくても**訪日ラボのジュニアエンジニアポジションとして面談推奨**とする。通常のエンジニアポジション（実務経験前提）とは判断基準が異なる点に注意

**「成長意欲重視」タグ付き求人の判断軸切り替え:**

求人タイトルや本文に以下のフラグがある求人は、*通常求人とは別の評価軸で判断する*。スキル軸（経験年数・職種経験の有無・技術スタック完全一致）で切ると的外れになる。

- *フラグの例*: 「【成長意欲重視】」「未経験OK」「ジュニア歓迎」「開発にも携わりながらPdM」「次世代の開発スタイル」「第二新卒歓迎」など、明示的に経験年数・職種経験のハードルを下げる表現
- 該当する求人例: 2101のうち「【成長意欲重視】インバウンド支援事業にて開発にも携わりながら次世代の開発スタイルを実践するPdMを募集＜生成AI活用＞」、訪日ラボのジュニアエンジニア求人など

判断軸を以下に切り替える:

1. *PdM経験／Rails経験／同職種経験の有無で切らない*。「経験が必要なポジションだから経験者しか面談しない」という発想は捨てる
2. *事業動機の解像度*：なぜインバウンド／なぜ当社プロダクト／なぜこの事業領域に移りたいかが言語化できているか（プロフィール上で読み取れない場合は「面談で確認」とする）
3. *自走力・アウトプット実績*：ハッカソン受賞、OSS活動、個人開発、技術ブログ、登壇、副業／インターン経験、ピアラーニングなど、自分で動いた実績の有無
4. *地頭・学習速度*：学歴（会津大／MARCH／関関同立／情報系の有名大学卒）、技術トレンドのキャッチアップ速度、コントリビューション数
5. *コミュニケーション力*：チームでの協業実績（多国籍チーム協業、ステークホルダー折衝、現場ヒアリング、合意形成）
6. *AI活用*：Claude Code/Cursor/Codex等を日常活用し、開発生産性を高めているか

これらの軸が揃っていれば、たとえ「PdM経験ゼロ」「Rails経験ゼロ」「現在の興味領域が当社事業と違う（例: ブロックチェーン特化）」であっても、*基本面談推奨*とする。「事業動機の解像度」「なぜ当社か」は面談で確認するものであり、プロフィール段階で否定材料にしない。

通常求人と成長意欲重視枠の見分けがつかない場合は、`hr/positions/` 配下の該当求人ファイルの本文を読んで、必須スキルが緩いか・経験年数指定がないかを確認する。

**日本語力の要件（全ポジション共通）:**
- 全求人ポジションで「ビジネスレベル以上の日本語力」が必須。ものづくりにおけるコミュニケーションの性質上、相当流暢な日本語が話せることが前提となる
- 日本語力の記載がない・外国籍の場合は、他の条件にかかわらず**基本お見送り**と判断する
- 英語話者を受け入れるキャパがない
- 例外的に面談を推奨する場合は、その根拠（日本在住歴・日本語資格・日本語での発信実績等）をプロフィールから明示すること
- **ただし現職または直近の職歴が日系企業の場合は考慮の余地あり**。日本語力の判断材料の一つとして加味する

JLPT 取得状況による判定:
- *JLPT N1*: ビジネスレベルに到達している可能性あり。他の条件次第で面談検討可
- *JLPT N2 以下*: ビジネスレベル未満。他の条件にかかわらず**基本お見送り**（N2 は中級レベルであり、当社のものづくりコミュニケーションには不足する）

### Step 4: 出力

**出力形式はSlack mrkdwn形式とする。** ユーザーがそのままSlackに貼り付けて使えるように、コードブロック（```）で囲んで出力する。

Slack mrkdwn の書式ルール：
- 太字: `*テキスト*`（アスタリスク1つ）
- 見出し相当: 太字テキストで代用（Slackに見出し構文はない）
- リスト: `•` または `-` で箇条書き
- 区切り線: `---`
- 絵文字: `:white_check_mark:`, `:x:`, `:warning:` 等を活用
- Markdownの `##`, `**テキスト**`, `- [ ]` 等は使わない

```
*スクリーニング結果: {候補者名}*

*候補者サマリー*
（候補者の経歴・スキルの要約を2-3行で）

---

*{ポジション名}*
:white_check_mark: *面談推奨: Yes* / :x: *面談推奨: No*

*必須スキル:* （充足状況を簡潔に）
*歓迎スキル:* （該当するものがあれば）
*人物像:* （推測できる範囲で）

*強み*
• （このポジションで即戦力となれる点、市場での希少性など）

*懸念点*
• （経験の不足、未知数な領域など。面談で深掘りすべきポイントとして記載する）

*推奨指名金額*（転職ドラフトの場合のみ。面談推奨: Yes の場合）
• 求人票レンジ: {ポジション}は{下限}〜{上限}万円
• 希望年収: {候補者希望}万円

**返信文章の生成について:**
- *候補者への返信文章は、ユーザーから明示的に依頼された場合にのみ生成する*（「返信文章を生成して」「返信して」「メッセージ書いて」など）
- *面談推奨: No の場合は、基本的に返信を送らない運用のため返信文章を生成しない*。お見送り判断の場合、候補者に返信せず選考段階を「お見送り」または「継続アプローチ」に変更するだけで終える運用。ユーザーから明示的に依頼された場合のみ、お見送り返信文を生成する
- 面談推奨: Yes の場合でも、スクリーニング結果のみを求められたときは返信文章を生成しない。返信文章は別途依頼されたときに生成する

**Findy Enterprise固有のルール:**
- 候補者ページの「いいかも受け取り済み」は**候補者から当社への「いいかも」を当社が受け取った**ことを意味する。当社から候補者に送ったわけではない
- 出力時に「いいかも送信済みポジション」のような当社発の表記をしない。単にポジション名のみ記載する

**転職ドラフト固有のルール:**

転職ドラフト（`job-draft.jp`）はカジュアル面談ではなく*指名（年収提示込みのスカウト）*を送るプラットフォーム。面談推奨: Yes の場合は、出力に *推奨指名金額* セクションを必ず含めること。

*推奨指名金額の算出ロジック*：以下の3要素を統合して、レンジ（最低/推奨/上限）を提示する。

1. *求人票の給与レンジ*: 該当ポジションの給与レンジを上限・下限の枠として使う。レンジ外の提示は基本しない（求人票更新の話は別途）
2. *候補者の希望年収*: プロフィール記載の希望年収を満たす（または近い水準まで寄せる）。希望年収が未入力の場合は、他社からの提示年収レンジ（直近6回の指名一覧から読み取れる）を参考にする
3. *市場価値*：
   - *経験年数 × 領域の専門性*: 同領域連続経験が長いほど市場価値が高い（Rails 8年 等）
   - *所属企業のブランド*: メガベンチャー／著名SaaS／大手（Sansan・LINEヤフー・Yahoo!・DeNA等）出身は基本+α
   - *学歴*: 旧帝大・早慶・MARCH関関同立以上は地頭の指標として加点要素
   - *他社競合状況（最重要）*: 直近回での指名件数・1位指名の有無・他社提示金額レンジ
     - 指名10件超・他社1位指名あり → 競争激化。レンジ上限近くで提示しないと埋もれる
     - 指名数件・他社辞退多数 → 標準的に提示、伸ばす必要は薄い
     - 提示年収レンジが見える場合（例: 825〜1,005万円）、当社は中央値〜上位25%を狙うと指名承諾されやすい
   - *アウトプット実績*: OSS活動・登壇・カンファレンス運営・コンペ入賞などは加点
   - *マネジメント・テックリード経験*: チームリード以上の経験は+α、CTO・EM経験はさらに+α

*出力フォーマット*：

```
*推奨指名金額: {下限}〜{推奨額}万円*
• 求人票レンジ: {ポジション}は{レンジ}
• 希望年収: {候補者希望}万円（or「未入力（他社提示レンジ {X}〜{Y}万円」）
• 市場価値: {他社からの指名状況・所属企業・経験・実績の要約}
• 推奨理由: {なぜこの金額か。求人レンジ・希望・市場価値のどれを優先したか}
```

*重要な注意点*：
- 当社の給与レンジを超える指名は基本しない。市場価値が当社レンジを超える場合は「指名は控えるか、求人レンジの上限ギリギリで挑戦」を提示する
- *希望年収の3割増し以上は不要*。希望以上に提示しすぎると、入社後の昇給余地が消える
- 転職ドラフトは期間内であれば指名件数に制限がないため、「指名枠の費用対効果」「優先度を下げる」「指名する価値があるか」のような枠の制約に関する言及は出力に*含めない*。承諾確率の見立てを述べるのは構わないが、枠を理由に指名を躊躇させる表現は避ける

## 注意点

- プロフィール情報は限られているため、断定的な判断は避ける。「面談で確認すべき点」として不明点を挙げるのは良い
- カジュアル面談のハードルは低めに設定する。少しでも可能性があれば面談を推奨する方向で判断する
- 候補者の個人情報は出力に必要最小限のみ含める
- **「面談で確認すべきポイント」はプロフィールに明記されている情報のみを根拠にする。プロフィールに書かれていないことを推測で確認ポイントに含めない。** 推測であることを記載する場合は「〜と思われるが確認」のように根拠が推測であることを明示すること
- **技術用語は正式な表記を使う。** 例: rspec → RSpec, vue.js → Vue.js, ruby on rails → Ruby on Rails, elasticsearch → Elasticsearch, typescript → TypeScript。候補者の経歴に記載された表記が不正確でも、出力時は正式表記に修正すること
