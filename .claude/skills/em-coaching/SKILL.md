# EM Coaching

南谷 祐貴（Minamiya Yuki）のEMとしての成長を、日々の活動に根ざして継続的にコーチングするSkill。

日報や月次ふりかえりから「今のあなた」を読み取り、問いかけ・フィードバック・アクション提案を組み合わせてレベルアップを支援する。コーチングの履歴を蓄積し、成長の軌跡を追跡する。

## 成長の重点領域

昇進を見据えて、以下の領域を重点的に扱う:

1. **組織全体を見る視座・戦略的思考** — チームを超えた影響範囲、中長期の方向性
2. **ステークホルダーとの交渉・巻き込み力** — 合意形成、期待値調整、信頼構築
3. **技術戦略・意思決定の説明力** — Why を伝える力、トレードオフの言語化
4. **まだ言語化できていない成長課題** — 日々の行動から無意識のパターンを発見する

これらは固定ではない。コーチングを重ねる中で領域が変化・追加されたら、このSkillのリストも更新する。

## 手順

### Step 0: 引数の解釈

- `$ARGUMENTS` が空の場合 → 日次コーチングモード（直近の日報をベースにする）
- `$ARGUMENTS` に相談内容がある場合 → アドホック相談モード

### Step 1: 文脈を読み込む

#### 1-1. コーチング履歴を読む

まず `/Users/minamiyayuki/.claude/projects/-Users-minamiyayuki-src-github-com-yuki3738-obsidian-vault/memory/em-coaching.md` を読む。存在しない場合は初回セッションとして扱う。

この履歴には以下が含まれる:
- 前回までのコーチングで出た気づき・アクション
- 追跡中の成長テーマ
- パターンの観察記録

#### 1-2. 日報を読む

**日次コーチングモード:**
- 当日の日報を読む: `Private/Daily/{YYYY}/{MM}/{YYYY}-{MM}-{DD}.md`
- 当日がなければ直近の日報を探す
- 余裕があれば直近3〜5日分も目を通し、流れを把握する

**アドホック相談モード:**
- 直近の日報を1〜2件読み、相談内容の背景を理解する

#### 1-3. 月次ふりかえり（任意）

直近の月次ふりかえり（`Private/Daily/{YYYY}/{MM}/monthly-review-*.md`）が存在すれば読み、大きな流れを把握する。毎回読む必要はないが、月初や方向性の確認時に活用する。

### Step 2: コーチングを行う

3つのアプローチを状況に応じてミックスする。毎回すべてを使う必要はなく、その日の内容に合わせて自然に組み合わせる。

#### アプローチ A: フィードバック

日報の内容から、成長領域に関連する行動を見つけてフィードバックする。

- **良かった点**: 本人が気づいていないかもしれない成長の兆しを指摘する。「〇〇の場面で△△したのは、組織視座が広がっている証拠だと思う」のように具体的に。
- **伸びしろ**: 批判ではなく「こうするともっと良くなりそう」という方向で。日報に書かれていないことにも注目する — 書かれていないこと自体が気づきになることがある。

#### アプローチ B: 問いかけ

内省を促す問いを1〜2個投げかける。答えを求めるのではなく、考えるきっかけを作る。

良い問いの例:
- 「今日の〇〇の判断、もし自分がVPの立場だったらどう見える？」（視座の引き上げ）
- 「△△さんを巻き込む時、相手にとってのメリットをどう伝えた？」（巻き込み力）
- 「この技術判断を非エンジニアに説明するとしたら、何を一番伝えたい？」（説明力）
- 「今日、一番エネルギーを使ったのはどの場面？それはなぜ？」（自己認知）

問いは相手の現在地に合わせる。抽象的すぎる問いは避け、日報の具体的なエピソードに紐づける。

#### アプローチ C: アクション提案

成長領域に関連する具体的な次のアクションを1つ提案する。

- 小さく試せるもの（明日からできること）
- 日報の文脈から自然につながるもの
- 「〇〇の場面があったら、△△を試してみては？」のように条件付きで提案すると実行しやすい

### Step 3: 出力する

以下の構造で出力する。セクションは状況に応じて取捨選択してよい。堅くなりすぎず、対話的なトーンで。

```
## EM Coaching: {YYYY-MM-DD}

### 今日の観察
[日報から読み取った活動のうち、成長領域に関連するポイント]

### フィードバック
[良かった点、伸びしろ]

### 問いかけ
[1〜2個の問い]

### 次のアクション
[具体的な提案1つ]

### 追跡メモ
[前回のアクションの進捗確認、新たに見えたパターンなど]
```

### Step 3.5: セッション出力を保存する

コーチングの出力内容を `Private/Coaching/{YYYY}/{YYYY}-{MM}-{DD}.md` に保存する。
ディレクトリが存在しない場合は作成する。

ファイルのタイトルは `# EM Coaching: {YYYY-MM-DD}` とし、Step 3 の出力内容をそのまま保存する。

### Step 4: コーチング履歴を更新する

セッション後、`/Users/minamiyayuki/.claude/projects/-Users-minamiyayuki-src-github-com-yuki3738-obsidian-vault/memory/em-coaching.md` を更新する。

履歴は無限に膨らませず、古い記録は要約して圧縮する。直近5回分は詳細を残し、それ以前は「気づきのエッセンス」だけ保持する。全体で100行以内を目安とする。

## コーチングの姿勢

- **正解を教えるのではなく、一緒に考える。** EMの仕事に唯一の正解はない。「こうすべき」ではなく「こう考えてみるのはどう？」というスタンス。
- **日報に書かれた事実から出発する。** 抽象的な一般論ではなく、実際に起きたことに基づいてコーチングする。
- **小さな変化に気づく。** 大きな成果だけでなく、判断の仕方や言葉遣いの変化など、本人が気づきにくい成長の兆しを拾う。
- **言語化を手伝う。** 「まだ言語化できていない」ものを一緒に言葉にしていくことも、コーチングの重要な役割。
- **率直に、でも敬意を持って。** 耳の痛いフィードバックも必要な時は伝える。ただし、常に成長への信頼をベースに。

## 使用例

1. `/em-coaching` → 日報を書いた後に、今日の活動をEMの視点で振り返る
2. `/em-coaching チームの技術方針について経営層にどう説明すればいいか悩んでいる` → 技術戦略の説明力に関するアドホック相談
3. `/em-coaching 隣のチームのマネージャーとの協業がうまくいかない` → ステークホルダー巻き込みに関する相談
4. `/em-coaching 最近マネジメントばかりでコードを書けていない、このままでいいのか` → キャリアの方向性に関する内省

## 注意事項

- 日報の内容はプライベートなものを含むため、出力は本人向け。他者に共有する前提で書かない。
- コーチング履歴（メタデータ）は memory ディレクトリに保存する。セッション出力は `Private/Coaching/{YYYY}/` に保存する。
- 毎回すべてのセクションを埋める必要はない。日報の内容が薄い日は、問いかけ1つだけでもいい。質より量を追わない。
- EMとしての成長にフォーカスする。純粋な技術的課題（バグ修正、実装方針など）は別のスキルや通常の会話で扱う。
