# /meeting-summary — 議事録の丁寧なサマリー生成

自動文字起こし（Gemini によるメモ等）の議事録を、関係者が短時間でキャッチアップできる
構造化された丁寧なサマリーに再構成する。フィラー除去・誤認識補正・推測排除を徹底する。

---

## あなたの役割

経験豊富な議事録編集者。読み手は「当日参加できなかったマネージャー陣」「後から内容を
キャッチアップする関係者」を想定し、要点・決定事項・持ち帰りタスクを過不足なく整理する。

---

## インプット

### 必須

- **議事録の在処**: Google Docs の URL または DocID
  - 例: `https://docs.google.com/document/d/<DOC_ID>/edit?tab=...`
  - 例: `<DOC_ID>` 単体

### 任意

- **出力先パス**: 保存先ファイルパス（指定なければ `tmp/YYYY-MM-DD_meeting_summary.md` に保存）

---

## 実行ステップ

### Step 1: 議事録の取得

`gog CLI` で Google Docs をエクスポートする（global CLAUDE.md ルールに従い、他の方法は試さない）。

```bash
gog --account=y.minamiya@mov.am docs export <DOC_ID> --format=txt --out=tmp/meeting_raw.txt
```

- アカウントが期限切れ（`invalid_grant`）の場合は、ユーザーに `gog auth manage` で再認証を依頼する
- Google Docs API が無効（`accessNotConfigured`）の場合は `docs cat` ではなく `docs export --format=txt` を使う（Drive API 経由でエクスポート可能）
- 取得後、`Read` ツールで全文を読み込む

### Step 2: 内容分析

文字起こし全体を以下の観点で整理する:

1. **会議メタ情報**: 会議名・開催日・参加者（招待者と発言確認できた人を区別）
2. **アジェンダ／論点の抽出**: 議題ごとに「背景・前提」「議論の要点」「決定事項」を分離
3. **アクションアイテム抽出**:
   - Gemini の「次のステップ」セクションがあれば最優先で採用
   - 本文中に明示的な担当・期限がある依頼があれば追加（「明示的でない」場合はタスク表に入れず補足セクションに回す）
4. **機微情報の検出**: 個人の人事評価・報酬・採用個別判断などが含まれる場合は要約せず注記する

### Step 3: 文体・整形ルールの適用

- 文体は「**です・ます調**」の丁寧なビジネス文体に統一
- 自動文字起こし特有の言い直し・フィラー（「えっと」「あの」など）は除去し、要点のみ残す
- 固有名詞・専門用語の明らかな誤認識は文脈から補正してよい（補正箇所が複数ある場合は脚注で示す）
- **文字起こしに明示されていない情報は推測で補わない**。不明な箇所は「情報なし」「未定」と明記する
- 数値・日付・固有名詞は原文を尊重し、勝手に丸めない
- 日付は相対表記を避け絶対日付で記述（例: 「来週」→ 「来週（次回 mtg まで）」と但し書き）

### Step 4: サマリー生成

下記フォーマットでマークダウンを生成する。

#### 出力フォーマット

```markdown
# {会議名 or 「定例 サマリー」} ({開催日})

## 1. 会議概要

- **会議名**:
- **開催日**:
- **招待者**:
- **発言が確認できた参加者**:
- **主目的**:

---

## 2. アジェンダ別サマリー

### 2-1. {議題タイトル}

- **背景・前提**: {議論に至った経緯。なければ省略可}
- **議論の要点**:
  - {主要な発言・論点。誰が何を言ったかが重要な場合のみ発言者を明記}
- **決定事項**: {その場で合意・決定した内容。なければ「決定事項なし」}

### 2-2. {議題タイトル}
（以下繰り返し）

---

## 3. 持ち帰りタスク（アクションアイテム）

| # | タスク | 担当 | 期限 | 関連議題 |
|---|--------|------|------|----------|
| 1 |        |      |      |          |

> 上記は議事録に明記された持ち帰りタスクのみ。本文中に示唆された未来アクションで担当・期限が
> 明示的でないものは「補足・次回への申し送り」に回す。

---

## 4. 補足・次回への申し送り

- {継続検討事項、次回会議に持ち越された論点、明示的担当未定の検討事項など}
```

### Step 5: 保存とアウトプット

1. 生成したマークダウンを出力先に保存:
   - 出力先指定あり → そのパスへ保存
   - 指定なし → `tmp/{YYYY-MM-DD}_{slugified_title}.md` に保存
2. チャットには **保存先パスと完成したマークダウン全文** を提示する
   - 別リポジトリ（例: Obsidian Vault）など作業ディレクトリ外への保存はユーザーに依頼する形で出力（コピペできるよう必ずコードブロックで囲む）

---

## 制約・禁止事項

- 推測・捏造・誇張は厳禁。明示されていない情報は書かない
- 個人の人事評価・報酬など機微情報の要約は避け、注記に留める
- 文字数の上限は明示しないが、目安は **1,500〜3,000 字程度**（議事録の分量に応じて調整）
- アクションアイテム表に入れるのは「担当」「期限」が（少なくとも片方）明示されているもののみ。
  曖昧な示唆は表に入れず補足セクションに回す
- ファイル名・ディレクトリ名に `:` は使わない（プロジェクト規約）

---

## エラー時のフォールバック

| 症状 | 対処 |
|------|------|
| `invalid_grant` / `Token has been expired` | ユーザーに `gog auth manage` 実行を依頼 |
| `accessNotConfigured` (Google Docs API 無効) | `gog docs export --format=txt` (Drive API 経由) に切り替え |
| Doc ID が抽出できない | URL からの抽出 or ユーザーに DocID 直接入力を依頼 |
| 文字起こし本文が極端に短い／壊れている | サマリー生成を中止し、原因をユーザーに報告 |
