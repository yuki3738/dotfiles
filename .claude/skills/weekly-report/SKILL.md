# Weekly Report (Findy 4-step format)

[Findy 山田裕一朗氏の note 記事](https://note.com/yuichiro826/n/na3673744543d) で提案された
「AI時代の週報フォーマット」に基づき、南谷 祐貴の週報を生成する。

## 記事の核心ルール（必ず守る）

1. **①「印象的な出来事・気づき」と②「経営とマネジメントの気づき」もドラフトまで書く**
   - 「自筆でないと意味がない」原則は捨てる。本人が手を加える前提のたたき台を、本人の文体で書き切る
   - 「カレンダーの予定をそのまま拾ってきた」「Notion/GitHub の要約を並べただけ」の週報は禁止
   - 一次情報（顧客・候補者・チームメンバーとの対話／新しい技術・施策の体験）を最優先する。日報に書かれていない出来事を捏造しない
   - 文体は本人（南谷）の日報文体に揃える（次節「文体ガイド」を参照）
2. **③数字・進捗は AI が定量データを集計してよい**（PR・1on1回数・面談数など）
3. **④個人の振り返りも本人の文体でドラフトする**（今週の学び・来週変えること）

## 文体ガイド（南谷の文体で書く）

①②④をドラフトする際は、対象週の日報の文体をそのままサンプリングする。
書き手のクセを再現することが目的なので、以下の観察ポイントを **対象週の日報を読んだ上で**確認してから書く:

- 文末: 「〜だ/である」「〜してる」「〜だった」が混じるカジュアル寄り。週報でも敬体（です・ます）にしない
- 一人称: 「自分」「俺」（場面で混在）。週報内では一貫させる必要はない
- 短文を重ねるリズム。1文1意で改行しがち
- 「〜してよかった」「〜が刺さった」「〜が効いてる」など主観評価をストレートに書く
- 専門用語は原語のまま（EM・1on1・スカウト・スクリーニング・PR など）
- 不確かなことは「〜な気がする」「〜かも」で逃げを残す

書き出しのトーン例（過去の日報から想定される範囲）:
- 「今週は〜が一番デカかった」
- 「〜の話、改めて考えると〜」
- 「〜してて気付いたんだけど」

ドラフトしたあと、コメントで `<!-- ドラフト。本人の言葉で書き直してOK -->` を残す。

## 引数

`$ARGUMENTS` のフォーマット: `[<YYYY-MM-DD>]`（省略可）

- 省略時: 今日を含む週
- `YYYY-MM-DD`: その日付を含む週

## 手順

### 0. 対象週の決定

- 週の定義: **月曜始まり〜日曜終わり**（ISO 8601）
- `$ARGUMENTS` が空なら今日（`date +%Y-%m-%d`）を起点にする
- 月曜日と日曜日の日付を計算する（macOS の `date -v` で算出する例: `date -v-Mon -j -f %Y-%m-%d <today> +%Y-%m-%d`）
- 表示用の週ラベル: `YYYY-MM-DD（月） 〜 YYYY-MM-DD（日）`

### 1. 日報の読み込み

対象週の日報を全て読み込む:

- パス: `/Users/minamiyayuki/src/github.com/yuki3738/obsidian-vault/Private/Daily/{YYYY}/{MM}/*.md`
- 月をまたぐ場合は両月のディレクトリを参照する
- Glob → Read（並列）で取得する
- 存在しない日はスキップする（休日と判断）

### 2. GitHub 活動の収集（③ 数字用）

```bash
gh search prs --author=yuki3738 --created="{月曜}..{日曜}" --limit=100 --json title,url,state,createdAt,repository --sort=created
gh search prs --reviewed-by=yuki3738 --created="{月曜}..{日曜}" --limit=100 --json title,url,state,createdAt,repository,author --sort=created
```

- 自分が作成した PR がレビュー一覧に重複したら除外する

### 3. 素材抽出 → ①ドラフトの執筆

日報を横断的に読み、**「現場でしか拾えない一次情報」**を抽出する。
抽出した素材から、①「印象的な出来事・気づき」のドラフトを 2〜4 トピック程度書く。
箇条書きの羅列ではなく、本人の文体（前節「文体ガイド」）で短い段落として書く。

抽出対象（記事の趣旨に沿う優先順位）:

1. **候補者・採用面談での印象的な発言/反応**（カジュ面・選考面接の記録から）
2. **チームメンバーとの 1on1 で出た気づき・課題**
3. **新しい技術・ツール・施策の試行体験**（Claude Code skills、AI活用、技術検証など）
4. **障害・トラブルから得た学び**
5. **社外イベント・登壇・他社との対話で得た一次情報**

書いてはいけないもの（記事の禁忌）:

- カレンダー予定の羅列（「火曜10時に〇〇MTG」など）
- GitHub Issue/PR タイトルの羅列
- Notion ページのコピペ要約
- 日報に書かれていない出来事・発言・感想の捏造

ドラフトの構造（参考）:

```
### {出来事のテーマ（10〜20字）}

{2〜5文。出来事 → そのとき自分が何を感じた/考えたか → 派生した問い、まで書く}

— 出典: 日報 YYYY-MM-DD「セクション名」
```

各ドラフトの末尾に「— 出典」を必ず残す。本人が事実確認できるように。

### 3-2. ②経営とマネジメントの気づきのドラフト

①で書いた出来事から、EM/マネジメント視点の解釈を **本人の文体で**書く。
「①で起きたこと」→「組織・事業として何が読み取れるか」→「次の打ち手」の流れを 1〜3 段落でまとめる。

ガードレール:
- 一般論で終わらない（「〜が大事だ」だけでは弱い）。①の具体エピソードに紐づける
- 日報に書かれていない事実は持ち込まない
- 過剰に断言しない（「〜な気がする」「〜かも」も使ってよい）
- 結論を「来週なにを変えるか」まで一段落としてもよい（④と重複してOK）

### 4. 数字の集計（③ 用）

日報と GitHub 活動から以下を集計:

- 稼働日数（日報が存在した日数）
- PR 作成数 / PR レビュー数（自作 PR を除く）
- 1on1 実施回数とその相手（日報の 1on1 記載から）
- カジュアル面談・選考面接の件数
- ミーティング総数（概算）
- その他、本人の仕事に紐づく数字（採用パイプライン進捗、リリース件数、稟議承認数など、日報から拾えるもの）

### 5. ④個人の振り返りのドラフト

日報の「所感」「学び」「Todo」セクションを横断し、本人の文体で短く書く:

- 今週の学び（1〜3個。1行ずつでよい）
- 来週変えること（1〜2個。具体行動レベル）

長く書かない。記事の趣旨どおり「短くてよい」を維持する。

### 6. 週報ファイルの生成

以下のテンプレートで出力する。**①②④は本人の文体でドラフトを書き切る**。
ドラフト直後にコメントで `<!-- ドラフト。本人の言葉で書き直してOK -->` を残す。

### 7. ファイルの保存

- **保存先**: `/Users/minamiyayuki/src/github.com/yuki3738/obsidian-vault/Private/Weekly/{YYYY}/{MM}/weekly-{日曜YYYY-MM-DD}.md`
  - `{YYYY}` と `{MM}` は **日曜（週末日）の年月** を使う（月またぎの週は週末側に寄せる）
- 既存の `2026-WXX.md`（Sailboat形式）とはファイル名で区別する
- ディレクトリが存在しない場合は作成する（`Daily` と同じく `{YYYY}/{MM}/` 構造）
- 同名ファイルが既にある場合はユーザーに上書き可否を確認する

### 8. ターミナルへのプレビュー

生成した週報の本文（ドラフト済みの①②④＋集計済みの③）をユーザーに表示し、
**「ドラフトです。文体や内容を必要に応じて書き直してください」と一言添える**。

### 9. Git commit（自動 push はしない）

ドラフト状態のまま commit する（本人があとで上書きする前提）。
push はユーザーが手動で行うか、本人の書き直し後にまとめて行う想定なので、
このスキルからの自動 push は **行わない**。

```bash
cd /Users/minamiyayuki/src/github.com/yuki3738/obsidian-vault
git add "Private/Weekly/{YYYY}/{MM}/weekly-{YYYY-MM-DD}.md"
git commit -m "docs: draft weekly report {YYYY-MM-DD}"
```

push の要否はユーザーに確認する。

## 注意事項

- **ドラフトはあくまで素材**: 本人が手を加える前提。流暢に書きすぎて「もう手を入れる余地がない」状態にしない。短く、引っかかりを残す
- **捏造禁止が最重要**: ①②④を本人の文体で書く以上、ハルシネーションが本人の言葉として残ってしまう危険がある。日報・GitHub に存在しない出来事・発言・感情を絶対に書かない（[feedback_daily_report_hallucination.md](../../../projects/-Users-minamiyayuki-src-github-com-yuki3738-obsidian-vault/memory/feedback_daily_report_hallucination.md)）
- **DM・1on1の内容のプライバシー保護**: 1on1 の相手名は出してよいが、機微な発言内容はドラフトに転記しない（本人が判断する余地を残す）
- **南谷さん固有のロール**: EM（Engineering Manager）として、①の優先度は「採用候補者との対話・チームメンバーとの 1on1・AI活用/技術検証」になりやすい。記事の例（顧客・商談）に縛られず、自分の役割の一次情報を取りに行く
- **数字は出典を明確に**: PR数は GitHub CLI、1on1回数は日報、と根拠が辿れるようにする
- **既存 Weekly ファイルの形式を変えない**: `2026-WXX.md` の Sailboat 形式は別の振り返り用なので触らない
- **文体ガイドを毎回確認**: 対象週の日報を読んでから本人の文末・語彙・リズムを再確認してドラフトに反映する（テンプレ的な「です・ます」で書かない）
