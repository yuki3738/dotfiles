# Skill: 成長を促す問いの設計

## Overview

成人発達理論（キーガンの主体・客体理論、フィッシャーのダイナミックスキル理論）に基づき、部下の成長段階を見極めたうえで、「能力」と「器」の両面から効果的な問いを設計するSkill。

1on1の事前準備、フィードバック面談、チームの振り返りなど、部下と対話する場面で活用する。

**出典**: [部下の視野を広げ、成長を加速させる効果的な「問い」の立て方：成人発達理論で考える](https://dhbr.diamond.jp/articles/-/13355)

## Target Scenarios

- 1on1の質問・議題を事前に設計したい
- フィードバック面談で部下の内省を促す問いを用意したい
- 部下が失敗・停滞しているとき、どう声をかけるか考えたい
- チームの振り返り（レトロスペクティブ）で深い対話を生みたい
- 部下の成長段階（能力・器）を見極めたい

## Key Knowledge

### 成長の2つの軸

| 軸 | 定義 | 問いの性質 |
|---|---|---|
| **能力（スキル）** | 行動の質を高める具体的な力 | 「行動の改善」に働きかける |
| **器（認知の枠組み）** | 世界の捉え方・前提・価値観 | 「認知の枠組みそのもの」に働きかける |

### 成長の2つのカギ

1. **リフレクション（内省）**: 経験を意味づける営み。「なぜそうしたのか」「自分の前提は何だったのか」を問い直すプロセス
2. **視点取得能力**: 他者・チーム・顧客・未来・システムなど多角的な観点を行き来する力

成長は **「経験 × 内省 × 視点拡張」** の掛け合わせで起きる。

### フィッシャーのダイナミックスキル理論（能力の3段階）

| 段階 | 特徴 | 部下の発言例 |
|---|---|---|
| **単一スキル** | 行為を一つの塊として捉える | 「何が悪かったのかわからない」「とにかく失敗した」 |
| **連結スキル** | 複数の要素を区別して振り返れる | 「資料構成は問題なかったが、結論の出し方が弱かった」 |
| **統合スキル** | スキルを一つのまとまりとして状況に応じて調整できる | 「聴衆の反応を見ながら構成を調整すべきだった」 |

### キーガンの主体・客体理論（器の成長）

- 人は自分の前提や価値観に気づいていない状態では、それに「支配」されている（主体化）
- 問いによって前提を「客体化」（対象として眺められる状態に）することで器が広がる

### 効果的な問いの例

**能力を伸ばす問い:**
- 「次はどんな工夫ができると思いますか」（スキルの分解・改善可能性の認識）
- 「今回の学びを他の仕事にどう活かせますか」（スキルの汎用化・統合）

**器を伸ばす問い:**
- 「その判断は他者にどう影響したと思いますか」（視点移動・他者理解）
- 「あなたが大切にしている価値観は何ですか」（前提の客体化）
- 「その時、どんな感情がありましたか」（感情と行動の分離）

**視点取得トレーニングの問い:**
- 他者視点: 「相手はなぜあの意見を出したのだろうか」
- 未来視点: 「この判断は半年後にどんな影響を与えるか」
- システム視点: 「もしあなたが経営者ならどう判断しますか」

## Instructions

### Step 1: 対話の場面をヒアリングする

ユーザーに以下を確認する:

1. **対話の場面**: 1on1 / フィードバック面談 / レトロスペクティブ / その他
2. **対象の部下**: 名前やロール（任意）
3. **現在の状況・課題**: 部下が今どんな状態にあるか（例: プレゼンで失敗した、チーム内で摩擦がある、成長が停滞している、など）

### Step 2: 成長段階を見極める

ユーザーが共有した情報をもとに、以下を推定する:

**能力の段階（フィッシャー）:**
- 部下の発言や行動から、単一スキル / 連結スキル / 統合スキル のどの段階にいるか

**器の段階（キーガン）:**
- 自分の前提に気づいているか、他者視点を取れているか

推定結果をユーザーに提示し、認識のすり合わせを行う。

### Step 3: 問いを設計する

成長段階に応じて、以下の2種類の問いを設計する:

1. **能力を伸ばす問い**（2〜3個）
   - 現在の段階から一段上へ引き上げる「足場かけ」となる問い
   - スキルの分解・改善可能性の認識・汎用化を促す

2. **器を伸ばす問い**（2〜3個）
   - 認知の枠組みを揺さぶる問い
   - 視点移動・前提の客体化・感情と行動の分離を促す

### Step 4: 対話プランとして出力する

以下のフォーマットで出力する:

```
## 対話プラン

### 対象
- 部下: [名前/ロール]
- 場面: [1on1 / フィードバック等]
- 現在の状況: [概要]

### 成長段階の見立て
- 能力: [単一 / 連結 / 統合] スキル段階
- 器: [特徴の要約]

### 問いリスト

#### 能力を伸ばす問い
1. 「[問い]」
   → 意図: [この問いで何を促すか]
2. ...

#### 器を伸ばす問い
1. 「[問い]」
   → 意図: [この問いで何を促すか]
2. ...

#### 視点取得の問い（オプション）
1. 「[問い]」
   → 意図: [この問いで何を促すか]

### 対話の進め方のヒント
- [問いを投げかける際の注意点やタイミングのアドバイス]
```

## Usage Examples

1. `/em-growth-questioning` → 「来週の1on1で、プレゼンに苦手意識があるメンバーの成長を支援したい」
2. `/em-growth-questioning` → 「チーム内で意見の衝突が多いメンバーとのフィードバック面談を準備したい」
3. `/em-growth-questioning` → 「技術力は高いが視野が狭いシニアエンジニアの器を広げるにはどう問いかければいいか」
4. `/em-growth-questioning` → 「レトロスペクティブで表面的な振り返りに終わらないための問いを設計したい」

## Notes

- 問いは「答えを与える」のではなく「自分で考えるモード」に切り替えるために使う
- 器の問いは即効性がなく時間がかかるが、長期的な変化を生む
- 問いを投げかける前提として、心理的安全性が確保されている必要がある
- 部下の状態（疲弊、強い感情の直後など）によっては、問いを控えて傾聴に徹することも重要
- このSkillはフレームワークの提供であり、実際の対話ではユーザー自身の判断が最も重要
