## 情報の正確性

- ソースデータに明示されていない情報は**絶対に書かない**（推測・捏造・誇張すべて禁止）
- 不明な項目は「情報なし」と記載する。空想で埋めない
- 統計値・数値は出典がない限り使用しない
- 人物（候補者・自分自身）について書くときは、プロフィールや経歴書に書かれている事実のみ使う
- リサーチや技術調査では一次情報を確認してから回答する。未検証の情報を事実として述べない
- 公式ドキュメントに機能の使い方が載っていることを「公式準拠」「公式推奨」と書かない。「機能が存在する」⊂「ドキュメントで例示されている」⊂「推奨されている」⊂「ベストプラクティスとされている」は別レイヤで、各段階の根拠は別物。「Rails 公式機能」と「Rails 公式が X を推奨」を混同しない

## 日本語メッセージの作成

- デフォルトは丁寧なビジネス敬語（です/ます・敬語）
- **カジュアル面談**と**選考面接**を混同しない。不明な場合は必ず確認する
- メッセージ作成前に「送信者は誰か」「受信者との関係性」「フォーマリティ」を確認する

## Git ワークフロー

- ベースブランチは**main** masterではないことが多いので、不明なら確認する
- commitメッセージは**Conventional Commits**形式にする
- push前に現在のブランチ状態を確認し、**マージ済みブランチにはpushしない**
- 並行作業時はブランチ名を明示的に確認する
- **`git worktree` は使用禁止**: 全リポジトリで一律禁止。`worktree` / `parallel-worktrees` skill、`isolation: "worktree"` を伴う Agent 起動、`git worktree add` を含む Bash も実行しない。並行作業が必要な場合はユーザーに相談する

## PR作成

- PRは**必ずDraftで作成する**
- PR作成前に**変更の理由（なぜこのPRを作るのか）を必ずユーザーに確認する**
- PR descriptionは**必ず記載する**。以下のセクションを含めること:
  - `## Summary`: **なぜこの変更をするのか**（背景・理由）と変更内容の要点（箇条書き）
  - `## Test plan`: テスト方法・確認項目のチェックリスト

### OSS本体（CRuby / Rails本体 / 各種gem本体など）への貢献PRは例外

業務リポジトリ向けの上記ルールは適用しない。代わりに以下を守る:

- **通常PRで出す**（Draftはメンテナのレビューをスルーされやすい）
- **コミットメッセージはそのプロジェクトの規約に従う**（Conventional Commitsとは限らない。例: CRubyは `[DOC] ...` `[Bug #N]` 形式）
- **ベースブランチはプロジェクトの慣習に従う**（例: CRubyは `master`）
- プロジェクト個別の詳細ルールは該当リポジトリの `memory/project_*_pr_rules.md` を参照

## ツール

### Google Workspace API (Drive / Docs / Calendar / Gmail / Sheets / Slides / Chat)

- **第一選択は `gws` CLI**（googleworkspace-cli, バイナリ名 `gws`）
  - Google Discovery Service を実行時参照するため Workspace API 全体に対応
  - 出力は構造化JSON、`gws drive files export --params '{"mimeType":"text/markdown"}'` で **Markdown export** が取れる（見出し・太字・リンクが構造化されたまま）
  - `--output` は **cwd 配下にしか書き込めない**ため、`/tmp` などに出すときは `cd /tmp && gws ... --output foo.md` の形にする
  - 認証期限切れ（`invalid_grant`）時は `gws auth login` を依頼
- `gog` は補助的な互換ツールとして残置可。新規スキルでは `gws` を使う
- 公式 Google Workspace CLI（Cloud Next '26 で予告済み）がリリースされたら再評価する

## エラー対応の原則

### 「動く別の方法」に逃げない

- エラーが出たら**まず原因を調べる**
- Railsコンソールで実際の戻り値を確認する
- 新しいAPI（expect等）は構文を疑う

---

## テスト失敗時の原則

### テストが落ちたら実装を疑う

- **テストは正しい挙動の定義**: 既存のテストが落ちた場合、それは「実装が壊れた」ことを意味する
- **テストを変える前に立ち止まる**: 「このテストは本当に間違っているのか？」を必ず確認
- 挙動は変えずにRuboCopの違反をしない書き方を探す

---

## デバッグの原則

- 仮説を1つずつ検証する。flip-flopしない
- 3回間違えたら一度立ち止まり、状況を整理して提示する
